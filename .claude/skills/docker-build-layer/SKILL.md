# Docker Build Layer

## Critical
- Generate `Dockerfile` and `.dockerignore` for supported runtimes only (Python, Ruby, Node.js).
- Only act for new/changed layers or functions, never retroactively alter existing files.
- Refer to specific file paths only (e.g. `Dockerfile`, `.dockerignore`).

## Instructions

1. **Target Directory and File Creation**
   - Confirm the target directory for the new build layer/function exists and is writable.
   - Example destination: any new layer or function subdirectory in project (e.g., new folder under appropriate area — see project conventions; do not refer to unconfirmed existing paths).

2. **Generate `.dockerignore`**
   - Create a new `.dockerignore` file alongside the new Docker build target.
   - Example content:
```
logs/
*.zip
Dockerfile
.dockerignore
.git
*.tar.gz
*.DS_Store
```

3. **Generate Dockerfile**
   - Python example:
```dockerfile
# Autogenerated. Do not edit by hand.
FROM python:3.11-slim
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt --target /opt
LABEL org.opencontainers.image.source=https://github.com/example/project
WORKDIR /app
```
   - For Node or Ruby, adjust accordingly (use correct base images and commands).
   - Do not add AWS credential mounting or reference excluded or removed files/folders in these files.

4. **Validate Docker Build**
```sh
docker buildx build .
```
- Confirm successful build output or error details before commit.

## Examples

User: "Setup Docker for a new layer directory"

- Create `.dockerignore` and `Dockerfile` as above in the appropriate new subdirectory.
- Run `docker buildx build .` in that directory to check build success.

## Common Issues
- Unsupported runtime? Do not write any files.
- Build fails? Both files must exist and include only required patterns. Check logs for guidance.
